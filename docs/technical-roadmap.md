# 忆伴相册技术开发计划

版本：0.1  
硬件路线：RK3576 + 10 寸屏幕  
阶段：工程样机技术方案

## 1. 技术路线总览

忆伴相册的第一代工程样机采用 RK3576 主控 + 10 寸屏幕作为终端硬件基础。

这个路线比通用平板和树莓派更适合后续产品化，因为它具备：

- 更接近量产硬件的嵌入式平台形态
- 足够运行 Linux / Android 图形界面
- 适合常开式家庭终端
- 可接入麦克风阵列、扬声器、人体感应、物理按键和状态灯
- 内置 NPU，可逐步承载本地 AI 能力
- 适合做无摄像头、低打扰、长期插电的桌面终端

第一阶段不追求所有 AI 都本地化，而是采用“本地交互控制 + 云端 AI 能力 + 可逐步边缘化”的混合架构。

核心原则：

> 先把老人端体验闭环做顺，再逐步把隐私敏感、低延迟、常驻能力下沉到 RK3576 本地。

## 2. 目标硬件形态

### 2.1 主控

主控平台：RK3576 核心板或开发板。

建议配置：

- RAM：至少 4GB，建议 8GB
- 存储：至少 32GB eMMC，建议 64GB
- 网络：Wi-Fi + Ethernet，工程样机可优先 Ethernet 保证稳定
- 系统：Debian / Ubuntu / Buildroot / Android 均可评估，第一版建议 Linux
- 显示输出：MIPI DSI / eDP / HDMI，取决于屏幕方案
- 音频：I2S / USB Audio / 3.5mm codec
- GPIO：用于人体感应、物理按键、状态灯、麦克风硬件关闭检测

### 2.2 屏幕

屏幕：10 寸。

建议规格：

- 分辨率：1280x800 起步，优先 1920x1200
- 亮度：300 nit 以上，避免白天看不清
- 触控：建议支持，但老人端不依赖触控
- 玻璃：低反光更适合相册形态
- 显示方向：默认横屏
- 可视角度：尽量选择 IPS 广视角

10 寸是比较合适的尺寸：比 7 寸更适合老人看照片和文字，又不会像电视一样侵入。

### 2.3 音频

音频是核心体验，不应弱化。

建议：

- 2 麦或 4 麦麦克风阵列
- 支持回声消除 AEC
- 支持语音活动检测 VAD
- 前向或侧向扬声器
- 音量足够大，声音清楚温和
- 物理音量键或旋钮可选

麦克风不只是录音器件，而是产品信任的一部分。必须设计明确的麦克风关闭能力。

### 2.4 人体感应

面对面陪伴模式需要检测“老人坐到相册前”。

建议优先级：

1. mmWave 雷达  
   更适合检测静坐、微动和存在状态；适合判断老人是否坐在相册前。

2. PIR 红外  
   成本低，但对静止老人不够稳定，更适合粗略触发。

3. ToF / 距离传感器  
   可判断前方距离，但布置和成本需要评估。

第一版工程样机可以先用 PIR 或 mmWave 模块接 GPIO / UART，软件上抽象为 presence 状态。

### 2.5 物理按键

建议至少保留：

- 麦克风关闭键
- 音量加减
- 电源 / 休眠键
- 可选：家属留言播放键
- 可选：紧急呼叫键，但第一版可以暂缓

麦克风关闭键最好能做到硬件级断开或至少硬件状态可检测，不只是 UI 上显示关闭。

### 2.6 状态灯

状态灯用于建立信任。

建议状态：

- 待机轮播：无灯或极弱灯
- 面对面陪伴：柔和呼吸灯
- 正在听：明确但不刺眼
- 正在说：轻微变化
- 麦克风关闭：固定可识别状态
- 网络异常：低频提示

灯效不要做成智能音箱式强科技感，要像家居设备。

## 4. 软件技术栈建议

### 4.1 老人端

推荐方案：Linux + Chromium Kiosk + Web/PWA。

原因：

- UI 迭代快
- 方便全屏相册展示
- 前后端技术统一
- 适合早期样机验证
- 后续可封装为 Electron / Tauri / 原生 Qt

老人端技术：

- React / Next.js 或 Vite React
- WebSocket 实时状态通信
- Web Audio API 或本地音频服务桥接
- 全屏 Kiosk 模式
- 本地守护进程管理传感器和硬件状态

建议拆分：

- `device-ui`：老人端全屏界面
- `device-agent`：RK3576 本地守护进程

`device-agent` 负责：

- GPIO / UART 读取人体感应
- 麦克风按钮状态
- 状态灯控制
- 音频设备检测
- 启动浏览器 Kiosk
- 断网重连
- 与后端保持 WebSocket / MQTT 连接

### 4.2 后端

推荐：Python FastAPI。

原因：

- AI 编排方便
- 与语音、图片、向量检索生态匹配
- 适合快速迭代

后端模块：

- Auth / Family
- Device
- Photo
- Message
- Conversation
- Memory
- Summary
- Notification
- Permission

数据库：

- PostgreSQL
- pgvector
- Redis
- MinIO / S3 兼容对象存储

### 4.3 家属端

推荐：Web 端先行。

第一版可以和后端管理界面合并，不急着做 App。

技术：

- React / Next.js
- 移动端优先布局
- PWA 可选

家属端核心页面：

- 今日摘要
- 照片上传
- 留言
- 家庭记忆
- 设备状态
- 隐私设置
- 声音授权

### 4.4 AI 编排

第一版建议用云端模型快速验证。

AI 模块：

- 图片理解：生成照片描述、人物线索、场景标签、回忆引导问题
- ASR：老人语音转文字
- LLM：围绕照片和长期记忆对话
- TTS：系统声音回复
- Summary：生成子女端亲情摘要
- Safety：过滤敏感内容和禁止分享内容

模型调用要通过统一的 AI Gateway，避免业务代码直接绑定某个模型厂商。

## 5. RK3576 本地能力规划

RK3576 的 NPU 可作为后续边缘 AI 的基础，但第一版不要过度依赖 NPU。

### 5.1 第一阶段：不用 NPU 也能跑通

目标是跑通体验：

- Kiosk UI
- 照片轮播
- 音频采集
- 音频播放
- 人体感应
- 云端 ASR / LLM / TTS

### 5.2 第二阶段：本地语音前处理

本地化：

- VAD
- AEC
- 降噪
- 唤醒词
- 打断检测

这些能力比本地大模型更优先，因为它们直接影响老人端体验。

### 5.3 第三阶段：本地轻量 AI

可尝试：

- 本地照片缩略图分类
- 本地敏感状态判断
- 本地短命令识别
- 本地小模型兜底回复
- 本地最近记忆检索

### 5.4 不建议第一版做的事

- 在 RK3576 上强行跑完整大语言模型
- 一开始就把照片理解全部放本地
- 一开始就做复杂声音克隆
- 为了展示 NPU 而牺牲产品体验

RK3576 的价值首先是稳定的终端计算、显示、音频和传感器控制，其次才是逐步承载边缘 AI。

## 8. 开发阶段计划

### 阶段 1：RK3576 基础系统 Bring-up

目标：确认屏幕、网络、音频、传感器基础可用。

任务：

- 烧录 Linux / Android 系统
- 点亮 10 寸屏
- 配置横屏全屏显示
- 测试触控
- 测试扬声器
- 测试麦克风
- 测试 Wi-Fi / Ethernet
- 接入人体感应模块
- 接入状态灯
- 接入物理麦克风按键

验收：

- 设备开机后能进入全屏 UI
- 能播放照片轮播
- 能录音并上传测试音频
- 能播放 TTS 音频
- 人体感应事件能被本地 agent 读取
- 麦克风按键状态能被 UI 显示

### 阶段 2：老人端相册 UI

目标：把设备变成可演示的亲情相册。

任务：

- 全屏相册轮播
- 大字体时间和天气
- 新留言提示
- 正在听 / 正在想 / 正在说状态
- 麦克风关闭状态
- 网络异常提示
- 面对面陪伴模式 UI

验收：

- 老人不用触控也能看懂当前状态
- UI 不像后台系统，不像监控设备
- 10 寸屏上文字清晰
- 远距离 1.5 米可读关键提示

### 阶段 3：后端和家属端 MVP

目标：打通照片和留言。

任务：

- 家庭账号
- 设备绑定
- 照片上传
- 照片列表
- 留言发送
- 老人端拉取照片和留言
- 老人语音回复
- 家属端查看回复

验收：

- 子女上传照片后，老人端能展示
- 子女发送留言后，老人端能播放
- 老人语音回复后，家属端能收到

### 阶段 4：AI 相册理解

目标：让 AI 围绕照片接得上话。

任务：

- 图片理解任务队列
- 照片描述生成
- 场景标签生成
- 回忆引导问题生成
- 人物关系手工标注
- 对话时检索当前照片上下文

验收：

- AI 能围绕当前照片发问和接话
- 不确定人物身份时不会乱编
- 同一张照片多次对话能逐步积累记忆

### 阶段 5：语音对话闭环

目标：老人可以自然说话，设备能听、想、说。

任务：

- 音频采集
- ASR
- LLM 对话
- TTS
- 打断
- 面对面模式连续对话
- 唤醒词模式
- 隐私指令识别

验收：

- 老人坐到设备前可以直接说话
- 设备不要求每句话重复唤醒
- 老人打断时能停止当前播报
- “这段别发”“休息吧”等指令优先执行

### 阶段 6：亲情摘要

目标：把老人表达转译给子女。

任务：

- 对话片段整理
- 记忆条目生成
- 敏感内容过滤
- 摘要生成
- 子女端摘要展示
- 一键回复
- 摘要反馈回流

验收：

- 摘要不像监控报告
- 摘要能让子女知道怎么回应
- 老人不愿分享的内容不会推送

### 阶段 7：小范围家庭测试

目标：验证产品真实价值。

测试对象：

- 3 到 5 个家庭
- 每个家庭至少使用 1 到 2 周

观察指标：

- 老人是否愿意坐到相册前说话
- 哪类照片最能触发回忆
- 子女是否打开摘要
- 子女是否回复
- 老人听到回复后的反馈
- 面对面模式是否自然
- 麦克风和隐私提示是否被理解

## 10. 风险清单

### 10.1 音频体验风险

风险：设备扬声器播放时，麦克风难以准确识别老人插话。

应对：

- 优先验证 AEC
- 选择成熟麦克风阵列
- 降低 TTS 播放时的误唤醒
- 打断逻辑先做按键 / 触摸兜底，再做全语音打断

### 10.2 人体感应误触发

风险：老人路过时设备频繁进入面对面模式。

应对：

- 检测停留时间
- 结合距离阈值
- UI 只轻提示，不每次主动说话
- 夜间降低主动性

### 10.3 RK3576 系统稳定性

风险：开发板系统、音频驱动、显示驱动存在适配成本。

应对：

- 先选择资料完整的 RK3576 开发板或核心板
- 第一阶段只做 Linux Kiosk，不做复杂系统裁剪
- 用 systemd 管理自启动和崩溃恢复
- 保留远程日志和 OTA 机制

### 10.4 AI 编造家庭事实

风险：AI 把推测当事实，造成家庭误解。

应对：

- 记忆分为 confirmed / inferred
- 不确定时提问，不断言
- 摘要引用来源
- 子女和老人都可以纠正记忆

### 10.5 隐私误解

风险：面对面模式被理解成“设备一直在听”。

应对：

- 明确无摄像头
- 状态灯显示监听状态
- 麦克风物理关闭
- 家属不能远程监听
- 默认不推送原文
